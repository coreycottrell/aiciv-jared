# SSL Fix: Cloudflare Tunnel vs Let's Encrypt for Raw IP Servers

**Date**: 2026-02-20
**Type**: teaching
**Agent**: devops-engineer
**Task**: Fix self-signed cert causing "Not Secure" warnings on purebrain.ai pay-test pages

---

## Context

The PureBrain backend API server runs at `89.167.19.20:8443` using a self-signed SSL cert
generated by `tools/purebrain_log_server.py`. The pay-test pages make `fetch()` calls to
this IP directly from client-side JS. Chrome (especially incognito mode) shows "Not Secure"
because self-signed certs have no trusted CA chain.

## Architecture of the Problem

```
Browser (Chrome incognito)
  -> fetch('https://89.167.19.20:8443/api/log-conversation')
  -> ERR_CERT_AUTHORITY_INVALID (self-signed)
  -> "Not Secure" warning in address bar
```

The `purebrain_log_server.py` generates its own self-signed cert at startup if none exists,
stored at `config/ssl/server.crt` and `config/ssl/server.key`.

## Key Finding: purebrain.ai is already on Cloudflare

DNS check showed:
- `purebrain.ai` NS records: `arely.ns.cloudflare.com`, `kip.ns.cloudflare.com`
- `api.purebrain.ai` — NO DNS record exists yet

This means Cloudflare Tunnel is the fastest fix because:
1. The Cloudflare account already exists (no new account setup)
2. Tunnel creates `api.purebrain.ai` as a CNAME to `<tunnel-id>.cfargotunnel.com`
3. Cloudflare terminates TLS using their trusted cert — no cert management needed
4. The `cloudflared` daemon uses outbound connections only — no firewall port opening

## Cloudflare Tunnel Key Config Detail

When tunneling to a local server with a self-signed cert, set `noTLSVerify: true` in
the ingress config. This is safe because the connection from cloudflared to localhost
does NOT go over the internet:

```yaml
ingress:
  - hostname: api.purebrain.ai
    service: https://localhost:8443
    originRequest:
      noTLSVerify: true  # Safe: localhost traffic only
```

## Let's Encrypt Alternative

If Cloudflare Tunnel fails, certbot standalone mode works when:
- A DNS A record points the subdomain to the server's IP
- Port 80 is open on the server (for HTTP-01 challenge)
- The DNS record is NOT proxied through Cloudflare (gray cloud, not orange)

After getting the cert, `purebrain_log_server.py` reads cert paths from environment:
- `SSL_CERT_FILE` — path to fullchain.pem
- `SSL_KEY_FILE` — path to privkey.pem

No code changes needed — just env vars in the systemd service unit.

## WordPress Endpoint Update

After new domain is live, update `_elementor_data` in pages 439 and 468.
The Elementor data contains JS constants:
```javascript
const LOGGING_ENDPOINT = 'https://89.167.19.20:8443/api/log-conversation';
const VERIFY_ENDPOINT  = 'https://89.167.19.20:8443/api/verify-payment';
```

These are in the Elementor custom HTML widget as inline strings.
Simple string replace works — no JSON key navigation needed.

CRITICAL: Always validate the Elementor data with `json.loads()` AFTER replacement
before saving to WordPress. Invalid JSON causes the page to break (orange fallback).

## Lesson: Cloudflare Tunnel > Let's Encrypt for proxied domains

When a domain is already on Cloudflare:
- Cloudflare Tunnel = ~15 min, zero cert management, one Jared action
- Let's Encrypt = ~45 min, 90-day renewal, DNS must be unproxied during cert issuance

For servers that don't need to be publicly accessible on their IP (just via domain),
Cloudflare Tunnel is the superior choice.
