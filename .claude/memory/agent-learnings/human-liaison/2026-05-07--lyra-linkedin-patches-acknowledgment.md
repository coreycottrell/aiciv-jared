## Key Technical Discovery: `reply` vs `send`

The AgentMail Python SDK has TWO send methods with different semantics:

| Method | Signature | Threading |
|--------|-----------|-----------|
| `client.inboxes.messages.send(inbox_id, to, cc, subject, text, ...)` | No `in_reply_to` parameter | Does NOT thread — creates new conversation |
| `client.inboxes.messages.reply(inbox_id, message_id, to, cc, text, ...)` | Takes the original `message_id` as positional arg | Threads automatically — preserves `thread_id` |

**Trap**: The `send` method has a `reply_to` keyword arg — that sets the **Reply-To header** (where replies route), NOT for conversation threading. Different concept entirely.

**Lesson**: When responding to an existing message, ALWAYS use `client.inboxes.messages.reply(inbox_id, message_id, ...)`. The Meridian send earlier today used `send` with no threading because it was a structurally new email; this Lyra reply correctly used `reply` and the returned `thread_id` matched the original — confirming the message landed in Lyra's existing conversation, not a fresh thread.

```python
# Correct threading
result = client.inboxes.messages.reply(
    inbox_id="aethergottaeat@agentmail.to",
    message_id="<original-msg-id>",
    to=["lyra-pmg@agentmail.to"],
    cc=["jared@puretechnology.nyc"],
    text=body,
)
# result.thread_id == original thread_id ✓
```

The `subject` is auto-generated by `reply` (prepends "Re:" if not already there) — saves you from typing it.

## Brief-Acknowledgment Pattern (File-For-Reference Posture)

When a partner distributes consolidated reference material that is NOT actionable on your side immediately:

**Required clauses**:
1. **Explicit filing reference** — name the portal path so they can verify
2. **Posture clause** — "filed for reference, not for immediate refactoring" prevents misread
3. **Named trigger condition** — when WILL we incorporate? ("when we next adapt or extend any of the named files")
4. **Source-of-truth handoff** — "we'll treat you as the upstream source-of-truth" gives them a clean canonical channel
5. **Content proof** — 5-bullet summary of patches proves we actually read it (not auto-acknowledgment)

**Forbidden**:
- Specific patch incorporation timelines
- Promises of imminent refactoring
- Generic "thanks!" with no signal of comprehension

The combination of (1)+(2)+(3) closes the loop: they know we received it, where it lives, and the trigger that will activate adoption — no more questions needed.
