## 1. What app.purebrain.ai Does

`app.purebrain.ai` is a redirect endpoint that sends users to `portal.purebrain.ai`, which serves the admin/login portal for PureBrain.

- **URL**: `https://app.purebrain.ai`
- **Behavior**: HTTP 302 redirect to `https://portal.purebrain.ai`
- **Purpose**: Provides a clean "app" entry point for users who type `app.purebrain.ai`

The admin portal at `portal.purebrain.ai` is served by a local Starlette server on port 8097, reverse-proxied through nginx on port 8099, fronted by Cloudflare tunnel.

## 2. How *.purebrain.ai Wildcard Works

### DNS Layer (Cloudflare)
- **Wildcard CNAME**: `*.purebrain.ai` points to our Cloudflare tunnel endpoint
- **Tunnel ID**: `fa55839c-e753-4a96-935c-cc58cf24b4b8`
- **SSL**: Cloudflare handles SSL for all subdomains (free plan wildcard coverage)
- Individual customer subdomains also get explicit CNAME records created via `cloudflared tunnel route dns`

### Cloudflare Tunnel (cloudflared)
- **Ingress rule**: Wildcard `*.purebrain.ai` → forwards to `localhost:8099`
- The tunnel terminates SSL and forwards plain HTTP to nginx

### nginx Layer (port 8099)
- **Main config**: `/etc/nginx/conf.d/purebrain-main.conf`
  - `portal.purebrain.ai` → `http://127.0.0.1:8097` (admin portal server)
- **Customer portals config**: `/etc/nginx/conf.d/purebrain-customer-portals.conf`
  - Auto-generated by `subdomain_router.py`
  - Each customer gets a `server {}` block: `{subdomain}.purebrain.ai` → Witness container URL
  - Includes WebSocket upgrade headers (portal uses WebSockets for live terminal)
  - Uses `set $backend` pattern with external DNS resolver to avoid startup resolution failures
- **Catch-all**: Unknown subdomains get a 404 with "Portal not found" message

### Traffic Flow
```
User browser
  → https://keenjared.purebrain.ai
  → Cloudflare (SSL termination)
  → cloudflared tunnel
  → nginx:8099 (hostname-based routing)
  → https://keen-jared-sanborn.ai-civ.com (Witness container)
```

## 3. How Customer Subdomains Are Auto-Provisioned

When a customer completes the birth pipeline, Witness fires a `birth_complete` webhook to Aether. Here is the full auto-provisioning sequence:

### Trigger: Birth Complete Webhook
```
POST https://api.purebrain.ai/api/birth/webhook
Header: X-Witness-Secret: witness-secret-2026
Content-Type: application/json

{
  "event": "birth_complete",
  "human_email": "jared@puretechnology.nyc",
  "human_name": "Jared Sanborn",
  "civ_name": "keen",
  "container": "aiciv-12",
  "magic_link": "https://keen-jared-sanborn.ai-civ.com/?token=abc123def456"
}
```

### Auto-Provisioning Steps (< 5 seconds total)
1. **Validate** auth header + payload fields
2. **Derive subdomain**: `civ_name` + first name from `human_name` → `keenjared` (lowercase, alphanumeric only, no hyphens)
3. **Check duplicates**: If `keenjared` exists, append number → `keenjared2`, `keenjared3`, etc.
4. **Create Cloudflare DNS**: `sudo cloudflared tunnel route dns {tunnel_id} keenjared.purebrain.ai`
5. **Generate nginx config**: New `server {}` block for `keenjared.purebrain.ai` → `https://keen-jared-sanborn.ai-civ.com`
6. **Reload nginx**: `sudo systemctl reload nginx` (graceful, zero-downtime)
7. **Rewrite magic link**: `*.ai-civ.com/?token=abc` → `keenjared.purebrain.ai/?token=abc`
8. **Log birth**: Append to `logs/birth_completions.jsonl`
9. **Send customer email**: Brevo template 30 with branded `keenjared.purebrain.ai` magic link
10. **Notify Jared**: Telegram message with birth details

### Idempotency
- Duplicate webhooks (same email + container) return `{"ok": true, "duplicate": true}` (200)
- Safe to retry without side effects

### Key Code
- **Webhook handler**: `tools/purebrain_log_server.py` (the `/api/birth/webhook` route)
- **Subdomain router**: `tools/subdomain_router.py` (DNS + nginx management)
- **Routes database**: `/home/jared/purebrain_routes.json`
- **nginx configs**: `/etc/nginx/conf.d/purebrain-customer-portals.conf` (auto-generated)

## 5. How Magic Links Get Rewritten

### The Problem
Witness generates magic links on the `*.ai-civ.com` domain:
```
https://keen-jared-sanborn.ai-civ.com/?token=abc123def456
```

Customers should see a branded PureBrain URL, not a technical backend domain.

### The Solution
On `birth_complete` webhook, Aether rewrites the magic link domain:

**Input** (from Witness):
```
https://keen-jared-sanborn.ai-civ.com/?token=abc123def456
```

**Output** (sent to customer):
```
https://keenjared.purebrain.ai/?token=abc123def456
```

### How It Works
1. The nginx reverse proxy at `keenjared.purebrain.ai` forwards ALL requests to `keen-jared-sanborn.ai-civ.com`
2. The token parameter passes through unchanged
3. Witness container sees the request as if it came directly
4. Customer sees only the branded `purebrain.ai` domain

The `*.ai-civ.com` domain still works directly -- we just layer `*.purebrain.ai` on top as a branded frontend.

## 7. Bug Fixes Deployed to Sandbox-3 (2026-03-04)

Three critical bugs were fixed in the sandbox-3 payment flow during this session:

### Bug 1: sanitizeText Missing (Blank Screen)
- **Symptom**: Blank white screen after payment on sandbox-3
- **Root Cause**: `sanitizeText()` function was missing from page code (existed in sandbox-2 but not sandbox-3)
- **Fix**: Added sanitizeText function to page 1232
- **Impact**: Chatbox would crash on first message render without this function

### Bug 2: IIFE Scope Bug (Script Error)
- **Symptom**: JavaScript error preventing chatbox initialization
- **Root Cause**: Immediately Invoked Function Expression (IIFE) had incorrect scope/syntax in the deployed code block
- **Fix**: Corrected IIFE wrapper syntax

### Bug 3: Portal Button Not Polling (containerName null)
- **Symptom**: "ENTER [AI NAME]'S BRAIN STREAM" button stayed greyed out permanently
- **Root Cause**: `runBirthInit()` was removed from page code (OAuth moved to portal), but it was the function that set `payTestData.containerName`. Without containerName, `runPortalButtonWatcher()` had nothing to poll.
- **Fix**: Added seed sender logic that derives containerName from AI name + human name, sends to `/api/birth/start`, and uses server-authoritative container name from response. Portal button watcher now has the container name it needs to poll.

### Additional Fix: portal-status Endpoint
- **Symptom**: Portal status always returned "not ready" even after birth completed
- **Root Cause**: Old proxy was intercepting `/api/birth/portal-status/{container}` and forwarding to Witness instead of checking local birth completion log
- **Fix**: Portal-status now checks local `birth_completions.jsonl` FIRST, only falls back to Witness proxy if not found locally
