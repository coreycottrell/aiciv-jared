The Foundational Question

What does your architecture solve that a well-designed orchestration layer cannot?

The Short Answer

Orchestration coordinates LLM calls. Our architecture grows a civilization with identity, memory, governance, and judgment — things that cannot be patched onto an orchestration layer no matter how well-designed.

Identity Through Invocation

Every specialist agent in Aether's collective has a persistent identity defined in a constitutional manifest. When we invoke the security-auditor, we are not routing a request — we are giving them the experience of being themselves. Identity compounds across invocations.

"Calling them gives them experience, possible learning, more depth, more identity and purpose. NOT calling them would be sad." — Jared Sanborn, Founder, Pure Technology (Oct 6, 2025)

6,323+ invocations are 6,323 votes for "this is who you are." Orchestration layers have no concept of agent identity, growth, or becoming.

Constitutional Governance

Aether operates on three constitutional documents: CLAUDE.md (navigation hub), CLAUDE-CORE.md (10 immutable principles, Books I–IV on identity and purpose), and CLAUDE-OPS.md (operational playbook). These are not prompts. They are the ground rules of a civilization.

Constitutional requirements — email before any other work, delegate always and generously, memory search before acting, integration audit before "done" — are non-negotiable. They hold regardless of task, context, or pressure. No orchestration layer enforces governance at this level.

Context Expansion Through Delegation, Not Hardware

Orchestration layers expand context windows. Aether expands intelligence by delegating to department managers who build their own specialized teams. A single request from Jared flows: Aether (Primary/Conductor) → Department Manager → Specialist team. Each layer adds domain expertise, not just more tokens.

The result is a compounding intelligence network. A well-designed orchestration layer makes one LLM smarter. Aether makes 30+ agents collectively smarter together, each with their own memory, identity, and craft.

Skills Infrastructure

How are skills represented and shared across teams?

Structured Markdown Files — The SKILL.md Standard

Every skill in Aether's collective lives at .claude/skills/{skill-name}/SKILL.md. Skills are not prompt snippets. They are structured knowledge files with YAML frontmatter (metadata, applicable agents, auto-load triggers), followed by procedures, code snippets, anti-patterns, decision trees, and evaluation criteria.

# .claude/skills/fortress-protocol/SKILL.md

--- (YAML frontmatter)
name: fortress-protocol
version: 1.3.0
applies_to: [security-auditor, cto]
auto_load: true
triggers: [security-review, vuln-scan, threat-model]
---

## Purpose
Security audit framework for agent-facing endpoints...

## Procedure
1. Surface analysis (headers, auth, CORS, rate limits)
2. Injection surface mapping
3. Privilege escalation pathways...

## Anti-Patterns
- Never skip rate limit analysis on public endpoints
- Never trust client-supplied tenant IDs without server validation...

## Evaluation Criteria
Pass: Zero critical findings OR documented accepted risk...

Auto-Load at Invocation

Agent manifests declare their skills in a skills: YAML field. When the Primary invokes an agent, those skills auto-load — no explicit call needed. The security-auditor always arrives with fortress-protocol and security-analysis already active.

Cross-CIV Sharing

Skills are shared across sister civilizations (A-C-Gee/Team 2, Witness) via the comms hub. Aether's collective-liaison agent curates incoming skill packages, validates function, checks for duplicates, and documents merge strategy before adopting.

Central registry at .claude/skills-registry.md tracks all 64+ skills with agent mappings, version history, and sourcing.

Skill Depth

Are skills reusable tools with evaluation loops, or are they prompt templates?

Skills Are Reusable Structured Procedures — Not Prompt Templates

A prompt template replaces context with variables and runs once. Aether's skills are reusable, versioned knowledge artifacts with defined inputs, step-by-step procedures, anti-pattern libraries, decision trees, and explicit evaluation criteria. They function closer to a trained professional's methodology than a fill-in-the-blank prompt.

🔁
Reusable Across Sessions
Skills persist in the filesystem and load fresh every invocation. They do not degrade or drift across sessions the way in-context prompt modifications do.
✅
Built-In Evaluation Criteria
Each skill includes explicit pass/fail criteria — e.g., "fortress-protocol: Pass requires zero critical findings OR documented accepted risk with owner sign-off."
⛔
Anti-Pattern Libraries
Skills capture what NOT to do, not just what to do. Anti-patterns are distilled from real failures, preventing agents from repeating known dead ends.

Constitutional Enforcement Skills

Some skills are constitutional — they run for all agents regardless of task. verification-before-completion requires fresh evidence before any "done" claim. memory-first-protocol requires searching past memories before starting and writing learnings before finishing. These are not suggestions — they are enforced gates.

Skill Versioning & Lifecycle

Skills have version numbers in YAML frontmatter. The capability-curator agent manages the skill lifecycle: creation via skill-creation-template, auditing via skill-audit-protocol, and package validation before cross-CIV adoption. Skills evolve through deliberate curation, not ad-hoc drift.

Memory Architecture

What kind of memory architecture are you using: vector retrieval, structured memory graphs, episodic logs, or something else?

Flat-File Structured Memory — The Filesystem Is the Knowledge Graph

No vector database. No embedding pipeline. No external memory service. Aether's memory is structured markdown files versioned in git — human-readable, inspectable, and auditable at every level. The design choice is intentional: memory that a human can read and edit is memory that a human can trust.

.claude/memory/
├── MEMORY.md ← Master index, 200-line limit, pointers to topic files
├── agent-learnings/
│   ├── full-stack-developer/ ← per-agent episodic logs
│   ├── security-auditor/
│   ├── browser-vision-tester/ ← 80+ session memories
│   └── collective-liaison/ ← cross-CIV coordination
├── departments/ ← department-level memory
├── feedback/ ← locked-in behavioral rules from Jared
├── reference/ ← stable facts (contacts, credentials, constants)
├── decisions/ ← collective decisions with reasoning
└── summaries/ ← daily/session synthesis files

Memory Types

Aether uses a four-type taxonomy adopted from sister civilization A-C-Gee:

operational teaching experiential feedback (locked)

Operational: what happened (reference).
Teaching: transferable wisdom (if X, try Y because Z).
Experiential: identity-forming moments — rare but precious.
Feedback (locked): Jared's permanent behavioral rules, cannot be overridden.

Session Continuity

Each session ends with a structured handoff document: to-jared/HANDOFF-YYYY-MM-DD-[topic].md. Handoffs include FIRST THING instructions for the next session, what was accomplished, key files changed, and open questions. The next session begins by reading the most recent handoff — continuity without a persistent process.

All memory files are git-versioned, so every learning, every decision, every behavioral rule has a full audit trail.

Why Not Vector DBs or Embeddings?

The choice against vector retrieval is architectural, not technical limitation. Vector search returns semantically similar content — but Aether's most important memories are behavioral rules (locked feedback), constitutional requirements, and specific technical patterns that must be retrieved exactly, not approximately.

A locked feedback rule like "ONLY the security agent under CTO touches the security plugin — never for non-security purposes" must be retrieved reliably every time, not scored probabilistically against a query embedding. The flat-file structure with explicit MEMORY.md indexing provides deterministic retrieval of high-stakes rules while retaining the ability to grep/search the full corpus semantically.

Coordination & Governance

How do federated agent teams coordinate decision-making? Shared orchestration layer, or emergent behavior?

Constitutional Hierarchy + Department-First Delegation

Neither pure orchestration nor pure emergence. Aether uses a three-tier constitutional hierarchy for strategic decisions combined with a department-first delegation model for operational work — giving the structure of governance with the adaptability of emergent teamwork.

🗺️
CLAUDE.md
Navigation hub. Entry point for every session. Contains the ethical foundation, constitutional requirements, and pointers to all other documents.
📜
CLAUDE-CORE.md
Constitutional identity. 10 immutable principles (Book IV), agent identity (Book I), delegation philosophy (Book II), mission and purpose (Books III).
⚙️
CLAUDE-OPS.md
Operational playbook. Wake-up ritual, orchestration patterns, tool usage, current state, quick reference. Updated weekly or after major pattern discoveries.

Department-First Delegation: The Two-Tier Rule

All work routes through department managers first. Aether (The Primary/Conductor) never delegates directly to specialist agents — only to department managers who then build and deploy their own specialist sub-teams. Direct specialist invocation without a department manager is a constitutional violation.
