## Architecture Summary (Current State)

The birth/seed pipeline involves two distinct JSON files that Witness uses:

**File 1 — Naming ceremony** (`purebrain_*.json`):
- Generated by pre-purchase chatbox (`/api/log-conversation` endpoint)
- Session ID format: `purebrain_{timestamp}_{random}` (set in chatbox line ~6085)
- Contains full conversation history including the moment the AI chose its name

**File 2 — Post-payment Q&A** (`pb-post-*.json`):
- Generated by post-payment chatbox (`/api/log-conversation` endpoint, `source: 'purebrain-post-payment'`)
- Session ID format: `pb-post-{orderId}` or `pb-post-{timestamp}`
- Contains messages[1]=name, messages[3]=email, messages[5]=company, messages[7]=role, messages[9]=goal
- This is the file where all 5 spec changes apply

## Item 1: `session_uuid` — UUIDv4 Idempotency Key

### Spec Requirement
Add a UUIDv4 to every `pb-post` payload. Witness uses this as an idempotency key. Same UUID = rejected. Once-ever per run.

### Current State
The `pb-post-*` session is built in `logPayTestData()` (chatbox v44, line ~8481-8486):
```javascript
const logSessionId = payTestData.prePurchaseSessionId
  || ('pb-post-' + (payTestData.orderId || Date.now()));

const convPayload = {
  session_id: logSessionId,
  messages: allMessages,
  source: 'purebrain-post-payment',
  // ...
  metadata: {
    event: data.event || 'pay-test-flow',
    orderId: payTestData.orderId,
    phase: 'post-payment',
  },
};
```

There is NO `session_uuid` (UUIDv4) field in the `convPayload`. The session_id is used as an identifier but it is not a UUIDv4 — it is a string like `pb-post-4MA83119W1272721N`.

The Aether proxy (`/api/intake/seed`) already has a deduplication mechanism based on `uuid` field extraction:
```python
seed_uuid = payload.get('uuid') or payload.get('seed_uuid')
if not seed_uuid:
    # Falls back to hash of orderId:stage:session_id
```

But this fallback is hash-based, not a proper UUIDv4. Witness's spec requires a real UUIDv4.

### Gap
**Missing.** No `session_uuid` field is sent in the `pb-post` payload (convPayload or payTestPayload).

### What Needs to Change

**Location 1 — chatbox JS (primary fix)**:
In `logPayTestData()`, generate a stable UUIDv4 once per flow init and include it in `convPayload`. The UUID must be generated ONCE (at flow start) and reused across all subsequent `logPayTestData()` calls for the same session. It must NOT regenerate on each call or deduplication breaks.

```javascript
// In payTestData initialization (around line ~8410)
payTestData.sessionUuid = crypto.randomUUID(); // Generate once per flow

// In convPayload (logPayTestData)
const convPayload = {
  session_id: logSessionId,
  session_uuid: payTestData.sessionUuid,  // ADD THIS
  // ... rest of fields
};
```

**Location 2 — log server** (`tools/purebrain_log_server.py`):
The `/api/log-conversation` handler should pass through `session_uuid` as a top-level field when present. Currently the log server captures optional fields but `session_uuid` is not in the `optional_fields` list (line ~548-554). Add it.

**Note on proxy dedup**: The `_seed_seen` dedup in `/api/intake/seed` currently uses `payload.get('uuid') or payload.get('seed_uuid')`. Once Witness's spec is implemented, the UUIDv4 will be in the session-level data, not necessarily in a top-level `uuid` field. Confirm with Witness whether the UUID goes to the top-level or inside `metadata`. The proxy dedup should be updated to match.

## Item 2: `naming_session_id` — REQUIRED Field

### Spec Requirement
`naming_session_id` is a REQUIRED field that links the `pb-post-*` file to the companion naming ceremony file. Without it, birth is hard-denied (no fallback).

### Current State
There is NO `naming_session_id` field anywhere in the current chatbox or log server. The pre-purchase session ID is stored as `payTestData.prePurchaseSessionId` and is included in `convPayload` as `prePurchaseSessionId` — but this is not the same as `naming_session_id` in Witness's spec.

The pre-purchase session (naming ceremony) session ID format is:
```javascript
const sessionId = 'purebrain_' + Date.now() + '_' + Math.random().toString(36).substr(2, 9);
```
(chatbox v44, line ~6085)

This is stored in `window._pbPrePurchaseSession.sessionId` and copied to `payTestData.prePurchaseSessionId` at flow start (line ~10543).

So the data EXISTS — it's captured as `prePurchaseSessionId` — but it is NOT being sent under the field name `naming_session_id` that Witness requires.

### Gap
**Field name mismatch.** The naming ceremony session ID exists in the payload as `prePurchaseSessionId` but Witness requires it as `naming_session_id`. This is a hard-denial failure condition if not fixed.

### What Needs to Change

**Location 1 — chatbox JS** (`logPayTestData()` convPayload):
```javascript
const convPayload = {
  session_id: logSessionId,
  naming_session_id: payTestData.prePurchaseSessionId || null,  // ADD THIS (rename/alias)
  // ... keep prePurchaseSessionId too for backward compat
};
```

**Location 2 — chatbox JS** (payTestPayload in `logPayTestData()`):
Same — add `naming_session_id` to the `/api/log-pay-test` payload for completeness.

**Location 3 — log server** (`/api/log-conversation` optional_fields list):
Add `naming_session_id` to the fields that get passed through to the JSONL log entry.

**Critical edge case**: If the user arrives at the post-payment chatbox WITHOUT a pre-purchase session (e.g. direct navigation), `prePurchaseSessionId` will be null. In that case `naming_session_id` will be null and Witness will hard-deny birth. This is expected behavior per the spec — we should log a warning so we can diagnose such cases.

## Item 3: `metadata.civ_name` — AI Name Extraction Optimization

### Spec Requirement
Put the AI name in `metadata.civ_name`. This saves Witness a Haiku API call and is the most reliable extraction path.

### Current State — PARTIALLY IMPLEMENTED

The log server already has a partial implementation in the `/api/log-conversation` handler:
```python
# tools/purebrain_log_server.py lines 559-569
if session_id.startswith('purebrain_'):
    meta = log_entry.get('metadata')
    if isinstance(meta, dict):
        ai_name = meta.get('ai_name') or meta.get('aiName') or ''
        if ai_name and not meta.get('civ_name'):
            log_entry['metadata'] = {**meta, 'civ_name': ai_name}
```

This handles the **naming ceremony** sessions (`purebrain_*`) — it promotes `metadata.ai_name` or `metadata.aiName` to `metadata.civ_name`.

However, for the **pb-post** sessions, the current `convPayload.metadata` is:
```javascript
metadata: {
  event: data.event || 'pay-test-flow',
  orderId: payTestData.orderId,
  phase: 'post-payment',
},
```

There is no `civ_name` (or `ai_name`) in this metadata block, even though `payTestData.aiName` is available.

The log server's `civ_name` promotion code only runs for `purebrain_`-prefixed sessions, not `pb-post-` sessions.

### Gap
**Incomplete.** `metadata.civ_name` is being set for naming ceremony sessions (good) but NOT for the pb-post sessions where Witness reads it. The fix needs to go in the chatbox.

### What Needs to Change

**Location — chatbox JS** (`logPayTestData()` convPayload.metadata):
```javascript
const convPayload = {
  session_id: logSessionId,
  // ...
  metadata: {
    event: data.event || 'pay-test-flow',
    orderId: payTestData.orderId,
    phase: 'post-payment',
    civ_name: payTestData.aiName || '',   // ADD THIS
  },
};
```

Also verify `payTestData.aiName` is always populated before `logPayTestData()` is called (it is — it's set from the `aiName` URL param or pre-purchase session at flow init, around line ~10535).

**No change needed in log server** — the log server already passes through `metadata` as a top-level optional field.

## Item 4: Goal Question — Q5 so `messages[9]` is non-empty

### Spec Requirement
Add "What is your primary goal?" as Q5 so that `messages[9]` is non-empty. Witness reads `messages[9]` for the goal.

### Current State — ALREADY IMPLEMENTED

The current chatbox v44 already has Q5. From the code and memory:
- Q1: Name → messages[1]
- Q2: Email → messages[3]
- Q3: Company → messages[5]
- Q4: Role → messages[7]
- Q5: Goal → messages[9]

The goal question text in chatbox v44 (line ~9496-9501):
```javascript
`Here's the one that matters most.<br><br>` +
`If ${aiName} could only do <strong>one thing</strong> exceptionally well for you — ` +
`what would make the biggest difference in your work or life?`,
```

And `payTestData.primaryGoal = goal` is set and included in `onboardingMsgs` if present:
```javascript
if (payTestData.primaryGoal) {
  onboardingMsgs.push({ role: 'assistant', content: 'If your AI could do one thing exceptionally well for you, what would it be?' });
  onboardingMsgs.push({ role: 'user', content: payTestData.primaryGoal });
}
```

The 2026-03-05 memory confirmed: "FORMAT CONFIRMED MATCHING as of 2026-03-04 test" and the prior spec ack explicitly listed messages[9]=goal as matching.

### Gap
**None.** Q5/goal/messages[9] is already implemented and confirmed working.

### Caveat
One risk: the goal question fires AFTER `runBirthInit()` (the OAuth flow). If the user drops off before answering Q5 (e.g. gets stuck on OAuth), `messages[9]` will be empty. This is a UX/flow risk, not a code gap. Witness hard-denies if messages[9] is empty. The current flow order (Q4 → birth init → Q5) means there is a window where birth init fires before goal is collected — Witness would process a seed without messages[9] if goal is never answered.

**Recommendation**: Confirm with Witness whether the seed is evaluated at `questionnaire:complete` (after Q5) or at `birth:authenticated` (after OAuth but before Q5). If the latter, we may need to reorder or delay seed submission.

## Item 5: Port 8099 → 8200 — Seed Intake API

### Spec Requirement
Port 8099 is NOT externally accessible. Use 8200 (seed-intake API). Need to register a partner API key for PureBrain.

### Current State — ALREADY CORRECT FOR SEED INTAKE; 8099 STILL USED FOR BIRTH PIPELINE

In `tools/purebrain_log_server.py` (lines 78-80):
```python
WITNESS_BIRTH_BASE_URL = 'http://37.27.237.109:8099'  # fleet host
WITNESS_SEED_INTAKE_URL = 'http://178.156.229.207:8200'  # awakening VPS
```

Two different servers:
- **Port 8099** on `37.27.237.109` = Witness Hetzner fleet host (used for `/api/birth/start`, `/api/birth/code`, `/api/birth/status`, `/api/birth/portal-status`)
- **Port 8200** on `178.156.229.207` = Witness awakening VPS (used for `/intake/seed`)

The seed intake proxy (`/api/intake/seed`) ALREADY forwards to port 8200:
```python
resp = http_requests.post(
    f'{WITNESS_SEED_INTAKE_URL}/intake/seed',  # = http://178.156.229.207:8200/intake/seed
    json=payload,
    timeout=15,
)
```

The chatbox does NOT call port 8099 or 8200 directly. All calls go through `https://api.purebrain.ai` (production) which proxies server-side. There are no direct port references in the chatbox JS for seed intake.

### Gap — Partner API Key (OPEN ACTION)

**The port routing is correct.** However the spec mentions: "Need to register a partner API key for PureBrain."

Current state: The seed intake proxy sends the payload as-is with no API key header. If Witness has implemented partner API key authentication on port 8200, our proxy calls will fail without that key.

**What needs to happen (not code — operational)**:
1. Corey/Witness needs to provision a partner API key for PureBrain
2. Aether needs to add it to `.env` (e.g. `WITNESS_PARTNER_API_KEY=...`)
3. The seed intake proxy in `purebrain_log_server.py` needs to add the key as a header:
```python
resp = http_requests.post(
    f'{WITNESS_SEED_INTAKE_URL}/intake/seed',
    json=payload,
    headers={'X-Partner-Key': os.environ.get('WITNESS_PARTNER_API_KEY', '')},
    timeout=15,
)
```

**Sandbox-specific**: The chatbox for sandbox mode routes birth API calls to `https://89.167.19.20:8443` (Aether's own server) not directly to 8099. So the sandbox chatbox never touches port 8099 directly.

## Notes for CTO / Implementation Team

The most architecturally clean approach is to make these changes in a single chatbox version bump (v4.6) that addresses items 1, 2, and 3 together, then deploy once. Deploying them separately risks partial states where Witness sees some fields but not others.

The `naming_session_id` rename is the critical path item — it should be treated as a breaking change on Witness's side. Coordinate timing with Corey before deploying, as deploying `naming_session_id` without Witness being ready to read it is harmless, but if Witness has already started hard-denying on its absence, we need to deploy immediately.

The `session_uuid` generation must use `crypto.randomUUID()` (available in all modern browsers) with a fallback for older environments:
```javascript
const genUuid = () =>
  typeof crypto !== 'undefined' && crypto.randomUUID
    ? crypto.randomUUID()
    : 'xxxxxxxx-xxxx-4xxx-yxxx-xxxxxxxxxxxx'.replace(/[xy]/g, c => {
        const r = Math.random() * 16 | 0;
        return (c === 'x' ? r : (r & 0x3 | 0x8)).toString(16);
      });
```
